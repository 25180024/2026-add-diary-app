# 仕様書 — 日記アプリ

## 1. プロジェクト概要

個人が日々の出来事・気分・思考を記録するためのモバイル日記アプリ。
シンプルで大人が使いやすい Apple ライクなデザインを採用し、
外部サーバー（Firebase）にデータを永続化することで、端末を変えてもデータが失われない。

---

## 2. 目的

- 日常の記録習慣を手軽に継続できる環境を提供する
- 直感的な UI で、書くことへの心理的ハードルを下げる
- 将来的な機能拡張（認証・複数端末同期など）に耐えうる設計を整える

---

## 4. 機能仕様

### 4.1 画面一覧

| 画面名 | ルート | 概要 |
|---|---|---|
| 日記一覧（ホーム） | `/` | 日記の一覧表示・検索・新規作成への導線 |
| 日記作成 | `/new` | 新しい日記の入力・保存 |
| 日記詳細 | `/entries/[id]` | 日記1件の全文表示 |
| 日記編集 | `/entries/[id]/edit` | 既存日記の編集・保存 |

> 日記検索は日記一覧画面に内包する（専用画面は作成しない）

---

### 4.2 日記一覧画面

- **無限スクロール**: 1ページあたり20件を取得し、スクロール末尾に達したら次の20件を追加ロード（Firestore の `startAfter` カーソルを使用）
- **並び順**: 日付の新しい順（降順）固定
- **検索バー**: 画面上部に常時表示
  - 検索対象: タイトル、本文（部分一致）、日付範囲
  - 検索中は無限スクロールを停止し、フィルタリング結果を表示
- **今日のカード**: 画面上部に当日の日記作成への導線カードを表示
- **FAB（+ボタン）**: 画面右下に固定表示。タップで日記作成画面へ遷移

---

### 4.3 日記作成・編集画面

#### 入力項目

| フィールド | 種別 | 必須 | 詳細 |
|---|---|---|---|
| アイコン（気分） | 絵文字ピッカー | 必須 | プリセット絵文字から1つ選択（デフォルト: ☀️） |
| タイトル | テキスト入力 | 必須 | 1文字以上 |
| 本文 | テキスト入力（複数行） | 必須 | 1文字以上 |
| 日付 | 日付ピッカー | 必須 | 当日の日付を自動セット。手動変更可能 |
| 画像 | 画像選択 | 任意 | 最大5枚。カメラロール・カメラ撮影の両方に対応 |

#### バリデーション

- アイコン・タイトル・本文・日付がすべて入力されている場合のみ保存ボタンを有効化
- 画像は任意のため未選択でも保存可能

#### 画像操作

- 追加: カメラロールから選択 または カメラで撮影
- 削除: 選択済み画像をタップしてキャンセル
- 上限5枚に達した場合は追加ボタンを無効化

---

### 4.4 日記詳細画面

- 全フィールド（アイコン・タイトル・本文・日付・画像）を表示
- 画像は横スクロールまたはグリッドで表示
- 編集画面への導線（編集ボタン）を設置
- 削除機能（確認ダイアログ付き）

---

### 4.5 日記検索

- 日記一覧画面の検索バーから操作
- **タイトル・本文**: クライアントサイドでの部分一致フィルタリング
- **日付範囲**: 開始日・終了日を指定して Firestore クエリでフィルタリング
- 検索条件をクリアすると通常の無限スクロール一覧に戻る

---

## 5. データモデル

### Firestore コレクション: `entries`

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | `string` | - | ドキュメントID（Firestore 自動生成） |
| `icon` | `string` | 必須 | 絵文字1文字（例: `☀️`） |
| `title` | `string` | 必須 | タイトル |
| `body` | `string` | 必須 | 本文 |
| `date` | `Timestamp` | 必須 | 日記の日付 |
| `images` | `string[]` | 任意 | Firebase Storage の画像 URL（最大5件） |
| `createdAt` | `Timestamp` | 必須 | 作成日時（Firestore サーバータイムスタンプ） |
| `updatedAt` | `Timestamp` | 必須 | 更新日時（Firestore サーバータイムスタンプ） |

### Firebase Storage

- パス: `entries/{entryId}/{filename}`
- 形式: JPEG / PNG
- 1ファイルあたりの上限サイズ: 10MB

## 7. アーキテクチャ方針

### ディレクトリ構成（予定）

```
src/
├── app/                          # expo-router ルート
│   ├── _layout.tsx
│   ├── index.tsx                 # 日記一覧
│   ├── new.tsx                   # 日記作成
│   └── entries/
│       └── [id]/
│           ├── index.tsx         # 日記詳細
│           └── edit.tsx          # 日記編集
├── components/                   # 再利用可能な UI コンポーネント
│   ├── EntryCard.tsx
│   ├── EmojiPicker.tsx
│   ├── ImagePickerGrid.tsx
│   └── SearchBar.tsx
├── services/                     # 外部サービスとの通信層
│   ├── firebase.ts               # Firebase 初期化
│   └── entries.ts                # Firestore CRUD 操作
├── store/                        # 状態管理
│   └── entries.tsx
├── hooks/                        # カスタムフック
│   ├── useInfiniteEntries.ts     # 無限スクロール用
│   └── useEntrySearch.ts         # 検索用
└── types/
    └── entry.ts                  # Entry 型定義
```

### レイヤー設計

- **`services/`**: Firestore / Storage との通信をすべてここに集約。画面コンポーネントから Firebase SDK を直接呼ばない
- **`hooks/`**: データ取得・状態管理のロジックをカスタムフックに切り出し、UI と分離
- **`components/`**: 再利用可能な UI パーツを細粒度で管理

## 8. Firebase 設計方針

### オフライン対応

- Firestore の **オフラインキャッシュを有効化**（`persistentLocalCache` オプション）
- オフライン時でも閲覧・書き込みが可能。オンライン復帰時に自動同期

### 将来的な認証対応

- 現時点は認証なし（単一ユーザー前提でデータを保存）
- Firebase Auth（メール/パスワード または Google ログイン）を後から追加できるよう、以下の設計を維持する
  - `services/firebase.ts` で Auth インスタンスを初期化済みとしておく
  - Firestore のコレクション設計を `users/{uid}/entries/{entryId}` へ変更しやすいよう、サービス層に操作を集約する

### セキュリティルール（初期）

```
// 現時点は全読み書きを許可（認証追加時に厳格化する）
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /entries/{entryId} {
      allow read, write: if true;
    }
  }
}
```

## 9. UI デザイン方針

- **テーマ**: Apple ライク。余白を広く取り、タイポグラフィを重視したシンプルなデザイン
- **カラーモード**: ライト・ダーク両対応（端末のシステム設定に追従）
- **カラーパレット（ライトモード）**:
  - 背景: `#FAF6EE`（ペーパーホワイト）
  - テキスト: `#2B2A28`（インクブラック）
  - サブテキスト: `#8A8278`
  - アクセント: `#8B5E3C`（ブラウン）
  - カード背景: `#FFFFFF`
- **フォント**: システムフォント（San Francisco / Noto Sans）を使用。外部フォントは原則導入しない
- **アニメーション**: 過度なアニメーションは避け、React Native 標準の遷移を使用

---

## 10. テスト方針

### 現時点

- **ユニットテスト（Jest + jest-expo）**のみ実施
- テスト対象: サービス層（Firestore CRUD）・カスタムフック・バリデーションロジック
- Firebase は jest の `__mocks__` でモックしてテスト

### 将来的な移行先

- **コンポーネントテスト**を `@testing-library/react-native` で追加
- テスト対象を画面・UI コンポーネントまで拡大
- E2E テスト（Detox 等）の導入も検討

---

## 11. 開発方針

- **型安全**: TypeScript strict モードを維持。`any` の使用は禁止
- **コード品質**: ESLint（eslint-config-expo）を CI で実行。警告ゼロを維持
- **コミット粒度**: 機能単位で細かくコミット
- **ブランチ戦略**: `main` を安定ブランチとし、機能開発は feature ブランチで行う
- **環境変数**: Firebase の設定値（API キー等）は `.env` で管理し、リポジトリにコミットしない
- **パッケージマネージャ**: pnpm を使用。`npm install` や `yarn` は使わない

---
